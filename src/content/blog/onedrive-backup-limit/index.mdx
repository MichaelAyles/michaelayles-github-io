My laptop was running hot doing nothing, and the fan was loud enough that I finally went looking for why. I expected a memory leak, some process slowly eating RAM until the machine swapped itself to death. That is the boring shape these things usually take. What I found instead was worse, and quieter, and had nothing to do with memory.

OneDrive was trying to sync 2.46 million files.

Microsoft's own documented recommendation is no more than 300,000. I was at roughly 8x that. And this is not a number I invented to be unkind. Microsoft's restrictions and limitations page says performance issues can occur once you pass 300,000 items, even if you are not actively syncing all of them, and older hardware is explicitly capped at that 300,000-item supported limit. Past that line the client starts losing. It lags, it throws errors, it silently gives up, and it does all of this without telling you in any way you would notice.

Here is the part that made me put my coffee down. The whole reason I tolerate OneDrive, the whole reason most people do, is the backup. And the backup is exactly the thing that stops working first. At 8x over the limit, "it's fine, it's in OneDrive" is no longer a true sentence. The sync is lagging or erroring or stalled, and the little cloud icon is still sitting there with a tick on it. You can lose a disk, go to restore from the cloud, and discover the cloud copy stopped being current months ago. You would find out at the exact worst possible moment to find out.

## How I got here without trying

I did not do anything exotic to end up 8x over a limit I had never heard of. I fell into a default.

Windows pushes Known Folder Move very hard. It is the friendly prompt that offers to "back up your Desktop and Documents to OneDrive," and it shows up insistently the moment you sign in with a Microsoft account. Say yes once, or let a managed M365 setup say yes for you, and your Desktop and Documents now live inside the sync folder.

Now think about where developers keep their code. Visual Studio defaults new projects to Documents\Visual Studio. Arduino, MPLAB, Unity, and the CodeWarrior and KDS firmware toolchains that got me all scatter their working trees through Documents too. And a working tree is not a handful of source files. It is node_modules with tens of thousands of entries, it is the .git object store, it is build and bin and obj and venv and __pycache__, all of it regenerable junk that has no business in a cloud backup but is now sitting right in the sync path.

When I actually counted, the shape was almost comic:

| Folder | Files |
|--------|-------|
| Desktop | 1,524,695 |
| Documents | 902,687 |
| Firmware and old workspaces | ~27,000 |
| Everything else | &lt;6,000 |

Desktop and Documents alone were 99% of it, both redirected into OneDrive by Known Folder Move. A Desktop with 1.5 million files is not documents. It is build artifacts and dependency trees, sitting in the one place guaranteed to get synced.

## "Who even has 300,000 files?"

Fair question. That number sounds like a hoarder with a decade of jpegs. It is not. It is a Tuesday afternoon of normal dev work.

When I drilled into the worst offender on my machine, a single folder sitting on my Desktop, it turned out to be a clone of [nxpSDK](https://github.com/Masmiseim36/nxpSDK), a public NXP microcontroller SDK maintained by Masmiseim36 on GitHub. If you are doing any embedded work on NXP micros it is genuinely worth knowing about, Masmiseim36 has done the unglamorous work of collating the SDK headers, drivers, and toolchain support files that NXP scatter across their own site. One repo. Not my code. Not unique data. 28 GB and 900,000 files, fully re-clonable from GitHub in an afternoon, and it had been quietly contributing most of my OneDrive overrun the whole time. Removing it knocked 900k files off the count in a single delete.

That is the shape of this problem. It is not that you have too much stuff. It is that one SDK clone, one Unity project, one node_modules from a side project you started six months ago and never cleaned up, lands you in OneDrive's failure zone before you have noticed anything is wrong. A single `npm install` in a folder under Documents and you have just added somewhere between 30,000 and 150,000 files to a sync client that starts losing the plot at 300,000. Do that a few times and the maths catches up with you fast.

To put real numbers on it: I have three serious active dev projects sitting on my Desktop. A complex FPGA computer vision system including RTL source, an R5 safety MCU, A53 embedded Linux, PetaLinux build images, and a documentation site. A hardware compiler web app with frontend, firmware blocks, and tooling. And a firmware and regulatory document set for commercial vehicle systems. Together, they come to about 110,000 files before any cache-stripping, and about 39,000 after. Three projects, comfortably within the 300k limit, zero problem.

The nxpSDK clone was 900,000 files on its own.

But here is the part that matters more than either of those numbers. My Projects folder has 145 projects in it. Most are nowhere near as large as those three. Some are bigger. Strip out the nxpSDK anomaly and the remaining 144 projects still total to around 1.56 million files. The 300k limit was always going to get crossed. The clone just made it happen faster and more obviously.

That is what the failure mode actually looks like. Not one catastrophic project. Just a normal developer's working history, accumulated over a few years, where each individual project seemed fine and the aggregate quietly crossed a line that nobody told you existed. There was never a moment where something went wrong. The count just went up, the sync client fell behind, the tick kept showing green, and nothing told you.

## How many other people are quietly in this hole

This is the bit that turned it from "my annoying afternoon" into something I wanted to write down. I am not special. The trap is a stack of defaults, and the same stack is under a lot of machines. So here is a rough back-of-envelope, the kind with big error bars but a solid direction:

<OneDriveFunnel client:visible />

| Stage | Fraction | Survivors | Why |
|-------|----------|-----------|-----|
| Professional devs worldwide | base | ~30M | SlashData and others, ~27 to 30M |
| Primary OS is Windows | ~47% | ~14M | Stack Overflow 2024. Mac/Linux with OneDrive face the same trap, just lower default adoption rate |
| OneDrive KFM actively syncing | ~35% | ~4.9M | Ships on by default, pushed on consumer and managed M365 |
| Keep code in Desktop or Documents | ~40% | ~2.0M | IDEs default there: Visual Studio, Arduino, MPLAB, Unity, KDS |
| Over the ~300k file limit | ~60% | ~1.2M | One node_modules is ~30k files, a few projects blow past easily |

Central guess, about 1.2 million developers in this state, with Windows being the biggest single contributor because Known Folder Move is pushed hardest there by default. The underlying trap is platform-agnostic though. If you have OneDrive running on macOS or Linux and your project folders are in the sync path, the 300k limit hits you the same way. The rate is lower because OneDrive is not installed or configured aggressively on those platforms by default, but lower is not zero. Low end maybe 0.3 million total, high end maybe 3 million, with most of the uncertainty in two soft fractions: how many actually have the sync running, and how many keep code in the synced folders.

Now put a consequence on it. Take the baseline rate at which any given machine loses its local disk in a year, drive failure plus ransomware plus an unlucky rm -rf in the wrong directory, call it 2 to 5%. Apply it to 1.2 million people who believe they are backed up and are not. That is somewhere between 24,000 and 60,000 developers a year who lose real work they were certain was covered. Not "the laptop felt slow." The repo is gone, and the cloud copy you were trusting topped out months back.

## Why nobody sees it coming

The reason this persists is that every single layer hides it.

OneDrive shows you a green tick that says you are synced. The 300,000 limit does not appear anywhere in the interface you actually look at, it is buried in a support article you would only read after something had already gone wrong. And the failure mode is silence. Not a red banner, not a notification, just sync that quietly stops keeping up while continuing to look healthy. There is no moment where the system says "I have given up on 2 million of your files." You are simply, gradually, not backed up, and nothing tells you.

I only found it because the fan annoyed me. That is a terrible early-warning system.

## "Yeah, but my code is on GitHub"

Fair. That is better than nothing. But GitHub only knows about files you have committed and pushed, which leaves out work in progress that has not been committed yet, ignored files that never went into the repo, binary assets that git handles badly, and scratch files and local configs. GitHub is a collaboration tool that happens to hold your source history. It is not a backup, and it would tell you the same.

The more awkward point is about everything else on your machine. Even if every line of code you care about is safely pushed to a remote, the files generating your 300k overrun, the node_modules, the .git object stores, the build trees, are still sitting in your sync path. OneDrive cannot keep up with them, so it falls behind on everything in the queue. Not just the regenerable junk. Everything. The work archive you made because it felt important. The photos that ended up on your Desktop because you were in a hurry. The half-finished invoice sitting in Documents. None of those have a git repo. All of them are invisible to GitHub. And all of them are failing to sync because the same queue is backed up behind two million files you would happily delete if you knew they were there.

Your code may be fine. It is the stuff that has no backup plan at all that is getting quietly dropped.

## What to actually do

If any of this sounds like your setup, the order matters. Do not start by cleaning up, because cleanup touches files and you cannot yet trust that the cloud has a copy. Backup first.

1. Get an independent copy now. An external drive is best, because it is fully offline and immune to all of this. A zip of your real working folders to a second cloud that is not choking works too. The goal is one known-good copy that does not depend on the client that is currently failing.
2. Then cut the file count. Exclude or move the regenerable junk: node_modules, .git, build and bin and obj, venv and __pycache__, toolchain caches. On a lot of machines that alone drops you from millions back under 300,000 and makes OneDrive functional again. Or move Desktop and Documents back to real local folders and take them out of the sync path entirely.
3. Then, and only then, decide whether you even want OneDrive in the loop for your dev work. That decision can wait. The backup cannot.

And before any of that, if you just want to know whether you are in the 1.2 million, run one of these. They count every file in your OneDrive folder and tell you where you stand.

**Windows (PowerShell):**

```powershell
(Get-ChildItem $env:OneDrive -Recurse -File -Force -ErrorAction SilentlyContinue | Measure-Object).Count
```

**macOS / Linux / WSL (bash):**

```bash
find "${1:-$HOME/OneDrive}" -type f 2>/dev/null | wc -l
```

Pass your actual sync folder as an argument if it lives somewhere else, for example `~/Library/CloudStorage/OneDrive-Personal` on macOS. Without an argument it guesses `~/OneDrive`.

These will run for a few minutes on a large tree. If the number that comes back is anywhere near 300,000, the backup you think you have is not the backup you have. Go sort out an independent copy before you do anything clever with the rest of it.

The cloud icon will keep showing a tick the whole time. That tick is not a promise. It turns out it never was.
